# Study Guide: Kernel Functions and Threading in CUDA

## Module Introduction
This module covers the core execution model of CUDA. It explores how CUDA bridges the gap between host (CPU) code and device (GPU) code by utilizing thousands of parallel threads. By understanding the organization of these threads and the keywords that define how code is executed, you will learn the fundamentals of writing massive parallel applications.

---

## Deep Dives

### 1. Kernel Functions and SPMD
A **Kernel Function** is the fundamental unit of parallel execution in CUDA. It specifies the code to be executed by all threads during a parallel phase.

*   **SPMD (Single Program, Multiple Data):** CUDA programming is a classic implementation of the SPMD style. In this model, every thread executes the exact same piece of code (the kernel), but each thread operates on a different portion of the data, determined by its unique index.
*   **SPMD vs. SIMD:**
    *   **SPMD:** Parallel processing units execute the same *program* on multiple parts of data.
    *   **SIMD (Single Instruction, Multiple Data):** All processing units are strictly executing the *same instruction* at the exact same instant. 
    *   *Analogy:* Think of SPMD as a group of students (threads) all given the same worksheet (program). They each have a different row of numbers to add (different data). They don't necessarily have to be at the exact same math problem at the exact same second, but they are all following the same "add these numbers" instruction set.

### 2. CUDA Thread Hierarchy
To manage the massive scale of GPU cores, CUDA organizes threads into a three-level hierarchy:

1.  **Grid:** The collection of all threads generated by a single kernel launch.
2.  **Thread Blocks:** The grid is divided into an array of thread blocks. All blocks in a grid must have the same size.
3.  **Threads:** Individual execution units. A single block can contain up to 1,024 threads.

*   **Hardware Mapping:** 
    *   The GPU executes these blocks on its Streaming Multiprocessors (SMs). 
    *   CUDA threads are designed for high efficiency; they take significantly fewer clock cycles to generate and schedule than traditional CPU threads.

### 3. Thread and Block Indices
Every thread is assigned a unique ID, which is critical for mapping the thread to the data it should process.

*   **`threadIdx`:** A unique identifier for a thread within its specific block.
*   **`blockIdx`:** A unique identifier for the block within the grid.
*   **`blockDim`:** The number of threads contained within a block (usually a multiple of 32 for hardware efficiency).

**The Global Index Formula:**
To map a thread to a unique piece of data across the entire grid, we use the formula:
`i = blockIdx.x * blockDim.x + threadIdx.x`
This creates a unique "global index" `i` for every thread, allowing each thread to know exactly which array element it is responsible for.

### 4. Execution Configuration Parameters
When the host launches a kernel, it must tell the GPU how many threads to create. This is done using the configuration syntax: `<<<grid_size, block_size>>>`.

*   **`grid_size` (Number of blocks):** Calculated by `ceil(n / threads_per_block)`. This ensures there are enough blocks to cover all `n` elements of the data.
*   **`block_size` (Threads per block):** Explicitly set by the programmer (e.g., 256).
*   **Efficiency:** If the vector length is not perfectly divisible by the block size, you must use an `if(i < n)` check inside the kernel to prevent "out-of-bounds" memory access.

### 5. CUDA Function Qualifier Keywords
CUDA extends the C language with keywords that tell the compiler where a function will run and who can call it.

*   **`__global__`:** Declares a **Kernel**. It runs on the device (GPU) and is called from the host (CPU).
*   **`__device__`:** Declares a **Device Function**. It runs on the device and can only be called from other device or kernel functions. (No recursion/indirect calls allowed).
*   **`__host__`:** Declares a **Host Function**. The default in C. It runs on the CPU and is called from the CPU.
*   **Combined (`__host__ __device__`):** Tells the compiler to generate two versions of the function—one for the CPU and one for the GPU.

### 6. Complete CUDA Program for Vector Addition
A standard CUDA workflow for vector addition involves these steps:

1.  **Allocate Host Memory:** Create vectors in CPU memory and initialize them.
2.  **Allocate Device Memory:** Use `cudaMalloc()` to reserve space on the GPU.
3.  **Data Transfer (Host to Device):** Use `cudaMemcpy(d_A, h_A, ..., cudaMemcpyHostToDevice)` to move data to the GPU.
4.  **Kernel Launch:** Execute the kernel: `vecAddKernel<<<grid, block>>>(d_A, d_B, d_C, n);`.
5.  **Data Transfer (Device to Host):** Use `cudaMemcpy(h_C, d_C, ..., cudaMemcpyDeviceToHost)` to retrieve the results.
6.  **Cleanup:** Use `cudaFree()` to release GPU memory.

## Key Takeaways
*   **Hierarchy Matters:** Threads are organized into blocks, and blocks into grids. Using this structure properly is the only way to parallelize code effectively.
*   **Index Calculation:** Always master the `i = blockIdx.x * blockDim.x + threadIdx.x` formula; it is the "DNA" of every CUDA kernel.
*   **Device Memory is Separate:** You cannot access CPU memory pointers inside a GPU kernel; you must allocate specific GPU memory and transfer data explicitly.
*   **Efficiency First:** Thread blocks should ideally have dimensions that are multiples of 32 to maximize hardware resource utilization (warps).
*   **Safety:** Always use the `if(i < n)` condition in kernels to prevent illegal memory access when the data size isn't a perfect multiple of your block size.
